# Frameworks Cheatsheet — Phase 4

Compact prompts for each of the six frameworks. Full source material in `08-knowledge/resources/2026-05-19-industry-reimagination-frameworks.md`.

## Universal requirement: handle + distillation per signal

Every signal generated by any framework MUST carry **two layers** alongside its structured ID:

1. **Memorable handle** — 3-5 words capturing the idea (e.g., "Stripe-for-merchant-agent-protocols", "Wirecutter-for-niche, AI-citation-first")
2. **Plain-English distillation** — one sentence describing what the signal proposes, in language a smart founder would use (not framework jargon)

These render in HTML gate reviews as card title (handle) + first line (distillation). The structured fields (cites, entry_power, framework_specific) appear below. RULE-1 from edge-cases.md.

## Run order

Frameworks 4.1-4.4 and 4.6 run in **parallel**. Framework 4.5 (7 Powers mapping) waits for 4.1-4.4 to complete because it operates on their outputs.

## 4.1 Blue Ocean ERRC

**Prompt:**
> Extract the competitive factors the industry currently competes on from Phase 1 value chain + Phase 2 customer satisfaction scores. For each factor, apply the four actions. The **Create** column is the most valuable output — factors no one offers that Phase 2 jobs/pains imply customers would value.

**Required output per signal:**
- `action`: eliminate | reduce | raise | create
- `factor`: the specific competitive factor
- `phase2_evidence`: which job, pain, or workaround supports this action
- `incumbent_constraint`: why the industry hasn't done this (Phase 1 evidence)

**Pass criterion:** ≥1 Create factor with named Phase 2 evidence. Generic "create better customer experience" fails.

**Anti-pattern:** Create column without Eliminate/Reduce. Blue Ocean requires the trade-off — value innovation breaks the differentiation-vs-cost frontier; adding factors without removing is feature creep.

## 4.2 Aggregation Theory Fit Test

**Diagnostic (Ben Thompson's three conditions):**
1. Is the direct relationship with the end user currently fragmented or supplier-owned? (Phase 1: `customer_relationship_owner`)
2. Is marginal cost of serving an additional user near-zero? (Phase 3: tech unlock supports this?)
3. Is supply plentiful and either commoditized or organizable? (Phase 1 fragmentation + Phase 3 supply_side)

**Pass:** ≥2 of 3 conditions. **Fail:** force-fitting Aggregation produces weak signals. Write `Aggregation Theory does not fit because [reason]` and move on.

**Thompson 2024 AI overlay:** If marginal cost test passes via AI inference, flag the **margin-cost regression risk** — AI reasoning models reintroduce per-transaction cost. Mark `margin_cost_regression_risk: low | medium | high`.

**Required output per signal (if diagnostic passes):**
- `who_aggregates`: the proposed platform
- `users_drawn_by`: the value prop that pulls users from going direct
- `supplier_behavior_change`: what suppliers do once the aggregator exists
- `margin_cost_regression_risk`: per Thompson 2024 overlay

**Anti-pattern:** "Aggregate using AI" without naming who currently owns the user relationship being displaced.

## 4.3 Decoupling Candidates — Teixeira

**Prompt:**
> For each top-3 pain point per entity (from Phase 2 score = intensity × frequency), run the 3-test decoupling check.

**The 3 tests:**
1. Could we own just THIS activity and let the customer/supplier complete the rest with the incumbent?
2. Would they pay (money OR data OR attention) for just this activity stripped out?
3. Does the incumbent structurally need this activity to be inside their flow (else lose the customer)?

A pain that passes all three is a **strong decoupling candidate**.

**Required output per signal:**
- `pain_source`: dataset path (e.g., `phase2.customers.gifting-shoppers.pain[evaluate]`)
- `concept`: one-line, specific
- `incumbent_conflict_test`: why this creates an unresolvable conflict for the incumbent
- `tests_passed`: count of 3

**Anti-pattern:** Decoupling concepts the incumbent can simply absorb (add a feature). The conflict test is what makes decoupling defensible.

## 4.4 Counter-positioning — Helmer

**Prompt:**
> For each major incumbent in `incumbents`, walk the 5-step trap analysis.

**The 5 steps:**
1. What is the incumbent's load-bearing profit pool? (`incumbents.[name].business_model.profit_pool_source`)
2. What revenue/cost dependencies trap them?
3. What business model would force them to choose between hurting their existing model OR letting us win?
4. What's the timeline before they have to respond?
5. What's the worst case response they could do (separate brand, etc.)?

**Pass criterion:** Step 3 has a real answer — incumbent genuinely can't copy without self-harm. "Just move faster" fails.

**Required output per signal:**
- `target_incumbent`: name
- `structural_trap`: the specific dependency that traps them
- `our_business_model`: what we'd do that they can't copy
- `response_timeline`: months
- `worst_case_response`: their best plausible counter-move

**Anti-pattern:** "They won't respond because they're slow." Slowness isn't a structural trap; profit-model dependency is.

## 4.5 7 Powers Early-Stage Mapping

Runs AFTER 4.1-4.4. For each candidate signal from those frameworks, tag entry power + projected scale power.

**Helmer's 7 Powers:**
- **Scale economies** — unit cost falls with volume
- **Network economies** — value to user rises with other users (same-side or cross-side)
- **Counter-positioning** — incumbent can't copy without self-harm (THIS IS USUALLY THE ENTRY POWER)
- **Switching costs** — users locked in by data, integration, habit
- **Branding** — durable preference creates pricing power
- **Cornered resource** — exclusive access to a critical input (OTHER COMMON ENTRY POWER)
- **Process power** — embedded organizational know-how that's hard to copy

**At entry:** only counter-positioning OR cornered resource are typically available. Others are scale-dependent.

**Required output per signal:**
- `entry_power`: counter-positioning | cornered-resource | none-at-entry
- `scale_power`: which of the 7 will hold at scale
- `mechanism`: specific causal chain (e.g., "transactions → recommendation data → better recs → more users → more transactions" for network economies)

**Anti-pattern:** "Network effects" as a label without specifying same-side or cross-side and what flows between users. Power names without mechanisms fail validation.

**Counter-positioning expiry warning:** if entry_power is counter-positioning, the scale_power MUST be different (counter-positioning expires as incumbents acclimate). Both fields equaling counter-positioning is a validation failure.

## 4.6 Thiel's Secret — hypothesis generator, NOT a truth gate

**The core fix (RULE-2): the human is never asked whether a Secret is true.** A secret you can endorse on the spot isn't a secret — by definition you don't yet know the answer; that's why it's worth a venture. Asking "which of these is true?" produces a shrug or, worse, pattern-matching to what *sounds* plausible (which quietly reimports the triangulation bias). So the Secret is treated as a **generator**: it produces a venture concept and a falsifiable hypothesis, and the human's job is to decide which hypotheses are worth *testing* — not which are true.

### Grounded form (required)

Every Secret must take this shape — it kills the "we believe AI changes everything" slogan failure mode:

> This industry assumes **[orthodoxy]**. That held because **[structural reason it was true historically]**. Capability **[specific Phase 3 capability_seed / condition, dated]** makes it false as of **[date]**.

A Secret that can't be written in this form (no structural reason, no dated capability) is an opinion, not a secret — discard it.

### AI candidate generation prompt

> Generate 5-10 candidate secrets — load-bearing orthodoxies in [industry] that the Phase 3 capabilities may have just falsified. Use three angles:
> 1. What does this industry assume is *necessary* that a Phase 3 capability now makes optional? (Orthodoxies in Phase 1, tested against `capability_seeds`.)
> 2. What does everyone assume customers *want* that the Phase 2 pain data quietly contradicts?
> 3. What trade-press consensus statement contradicts the Phase 1-3 evidence?
>
> For each candidate, produce:
> - **handle**: 3-5 word memorable name (e.g., "Personalization-is-overrated-for-gifting")
> - **grounded form**: the "industry assumes X; held because Y; capability Z makes it false as of [date]" sentence
> - **the bet it generates**: the venture concept that wins if this secret is true — emitted into the first-principles lane (Move 8 or whichever move the secret implies)
> - **load_bearing_hypothesis**: the secret restated as a single falsifiable claim specific to that venture
> - **why_unknown**: why this can't be settled by desk research (usually a behavioural fact that doesn't exist in the data yet)
> - **validation_test**: `{experiment, cost, time_to_signal, pass_threshold, fail_threshold}` — the cheapest experiment that resolves the hypothesis
> - **value_if_true**: the prize if the hypothesis holds

### Human gate prompt (test-worthiness, NOT truth)

> Each of these secrets has produced a venture concept, a single load-bearing hypothesis, and the cheapest test that would tell you if the hypothesis holds. You are **not** being asked which secrets are true — you can't know that yet, which is the point.
>
> For each: given the prize if true (`value_if_true`), the cost and speed of the test (`validation_test`), and your strategic fit — **which of these experiments are worth funding?**
>
> [Show candidates with handle + grounded form + the generated concept + load_bearing_hypothesis + validation_test + value_if_true]

The selected bets carry into Phase 5 as first-principles-lane concepts. Conviction is the *output* of running the test, not an input to this gate.

### Required output per Secret

```yaml
- signal_id: thiel-secret-1
  framework: thiel-secret
  handle: string                 # 3-5 word memorable name
  grounded_form: string          # "industry assumes X; held because Y; capability Z makes it false as of [date]"
  generates_concept: string      # concept_id of the venture this secret emits (origin: capability-first)
  load_bearing_hypothesis: string  # the secret as a single falsifiable claim
  why_unknown: string            # why desk research can't settle it
  validation_test:
    experiment: string
    cost: string
    time_to_signal: string
    pass_threshold: string
    fail_threshold: string
  value_if_true: string
  phase_evidence_cites: [string] # capability_seed IDs + Phase 1-3 paths
```

**Anti-patterns:**
- Secret as marketing slogan ("we believe AI changes everything") — fails the grounded-form requirement (no structural reason, no dated capability)
- Asking the human "which of these is true?" — violates RULE-2; ask which tests are worth funding
- A Secret with no `validation_test`, or a test too slow/expensive to actually run — an untestable secret is a daydream, not a bet
- A Secret that doesn't emit a concept — Secrets generate ventures; one that doesn't is just commentary
